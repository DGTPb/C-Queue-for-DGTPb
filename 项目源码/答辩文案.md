# 云山湖景区地图导航系统 — 答辩文案

---

## 一、需求背景

随着旅游业的快速发展，景区规模越来越大、景点越来越多，游客在游览过程中常常面临以下问题：

- **寻路困难**：景区道路纵横交错，游客难以快速找到最优路线；
- **信息不对称**：缺少对景点分布、道路距离的全局感知；
- **游览效率低**：没有合理的游览顺序规划，容易走回头路。

传统的纸质地图只能提供静态信息，无法进行路径计算和智能推荐。因此，我们开发了这套**基于 Dijkstra 最短路径算法的景区导航寻路系统**，旨在为游客提供：最短路径导航、游览路线推荐、景区地图可视化等一站式服务。

---

## 二、功能点

本系统共实现 **5 大核心功能**：

| 编号 | 功能 | 说明 |
|------|------|------|
| 1 | **景区地图可视化** | 以 ASCII 字符画渲染景区地图，区分步道（`.`）和公路（`:`），支持路径高亮（`*`） |
| 2 | **景点列表与详情** | 展示全部景点信息（编号、名称、类别、坐标、简介），支持按编号查询 |
| 3 | **最短路径导航** | 基于 Dijkstra 算法计算两点间最短路径，输出路径序列、总距离、预估步行时间 |
| 4 | **推荐游览路线** | 基于最近邻启发式算法，为游客规划游览多个景点的最优顺序 |
| 5 | **数据热重载** | 运行时重新加载景点和道路数据文件，无需重启程序 |

## 四、数据结构

### 4.1 景点结构体 `Spot`

```c
typedef struct {
    int  id;              // 景点唯一编号
    char name[64];        // 名称
    char category[32];    // 类别（山景/水景/古迹...）
    char description[256];// 简介
    int  x, y;           // 地图网格坐标（列, 行）
} Spot;
```

### 4.2 道路结构体 `Path`

```c
typedef struct {
    int  from;            // 起点编号
    int  to;              // 终点编号
    int  distance;        // 距离（米）
    char type[16];        // 类型（步道 / 公路）
} Path;
```

### 4.3 图结构体 `Graph`（核心）

```c
typedef struct {
    Spot spots[MAX_SPOTS];         // 景点数组，最多 64 个
    int  spot_count;
    Path paths[MAX_PATHS];         // 道路数组，最多 256 条
    int  path_count;
    int  adj[MAX_SPOTS][MAX_SPOTS]; // 邻接矩阵
    // adj[i][j]: 0=自身, -1=不连通, >0=距离(米)
} Graph;
```

**为什么用邻接矩阵？**
- 景区规模较小（≤64 景点），邻接矩阵空间开销可接受；
- 查询两点间是否有边只需 O(1)，Dijkstra 松弛操作非常高效；
- 实现简洁，适合教学演示场景。

### 4.4 辅助数据结构

| 结构 | 用途 |
|------|------|
| `DijkstraResult` | 存储单条最短路径：下标序列 + 总距离 |
| `RouteResult` | 存储推荐游览路线：访问序列 + 总距离 + 预估时间 |
| `MapPath` | 地图高亮路径：景点下标序列 |
| `dist[]` / `prev[]` / `visited[]` | Dijkstra 算法运行时的临时数组 |

## 五、核心算法

### 5.1 Dijkstra 最短路径算法

这是本系统的核心算法，用于计算从起点到所有可达景点的最短路径。

**算法流程（对应代码 `graph_dijkstra`）：**

```
1. 初始化：dist[] = -1(∞)，起点 dist[src] = 0，visited[] = 0
2. 循环 n 轮，每轮：
   a. 在未访问点中，贪心选出 dist 最小的顶点 u
   b. 标记 u 为"已确定"（visited[u] = 1）
   c. 用 u 松弛所有邻居 v：
      若 dist[u] + adj[u][v] < dist[v]，则更新 dist[v] 和 prev[v]
3. 结束：dist[] 存储最短距离，prev[] 存储前驱链
```

**时间复杂度**：O(V²)，V 为顶点数。对于 V≤64 的景区图，性能完全满足实时交互需求。

**路径回溯**（对应代码 `graph_shortest_path`）：

```
从终点 dst 出发，沿 prev[] 反向追溯到起点 src，
用栈收集逆序路径，再反转为正序输出。
```

### 5.2 最近邻启发式游览路线推荐

用于解决"从起点出发，游览所有目标景点"的 TSP（旅行商问题）近似求解。

**算法流程（对应代码 `route_recommend`）：**

```
1. 从起点出发
2. 对当前点运行 Dijkstra，计算到所有未访问目标的距离
3. 选择距离最近的未访问目标作为下一站
4. 重复 2-3，直到所有目标都被访问
```

**特点**：贪心策略，虽然不是全局最优解，但在实际景区场景下效果良好，且计算速度快。

### 5.3 Bresenham 线段算法（地图渲染）

用于在字符网格上绘制景点之间的连线道路，支持：
- 普通道路渲染（`.` 和 `:`）
- 高亮路径渲染（`*`），跳过景点格和边框格

## 六、技术亮点

### 亮点 1：用 `-1` 哨兵替代 `INF` 常量

标准 Dijkstra 通常用一个大整数 `INF` 表示"不可达"，本系统用 `-1` 作为哨兵值：

```c
if (dist[v] < 0 || nd < dist[v]) {   // dist[v]==-1 等价于 ∞
    dist[v] = nd;
    prev[v] = u;
}
```

**优势**：避免 `dist[u] + w` 与 `INF` 相加时的溢出风险，代码更安全。

### 亮点 2：跨平台终端适配

```c
#ifdef _WIN32
    SetConsoleOutputCP(CP_UTF8);       // 启用 UTF-8
    SetConsoleMode(h, ENABLE_VIRTUAL_TERMINAL_PROCESSING); // 启用 ANSI 颜色
#else
    printf("\033[2J\033[H");           // POSIX 清屏
#endif
```

同一套代码在 Windows 终端和 Linux/macOS 终端均可正确显示中文和彩色输出。

### 亮点 3：ASCII 字符地图可视化

无需任何图形库，仅用字符在终端中渲染出一张完整的景区地图：
- `#` 边框、`.` 步道、`:` 公路、`*` 高亮路径
- 数字 `1-9` 和字母 `a-z` 标记景点
- 支持路径高亮，直观展示导航结果

### 亮点 4：数据与逻辑分离

景点和道路数据存储在外部文本文件（`spots.txt` / `paths.txt`）中，支持运行时热重载，修改数据文件后无需重新编译即可生效。

### 亮点 5：重边自动取最短

构建邻接矩阵时，若两景点间有多条道路，自动保留距离最短的一条：

```c
if (g->adj[ia][ib] < 0 || d < g->adj[ia][ib]) {
    g->adj[ia][ib] = d;
    g->adj[ib][ia] = d;
}
```

## 七、成果展示

### 7.1 数据规模

- 景区包含 **10 个景点**（游客中心、樱花径、听涛亭、明镜湖、古寺、飞瀑、索道站、云山主峰、观景台、露营地）

| 游客中心(1) | 露营地(10) | 1→2→3→4→10 | 2200m |
| 古寺(5) | 观景台(9) | 5→6→7→8→9 | 2200m |

所有路径均经手工验证正确。
